## Priority 1: Bug Fixes (hours)

Known bugs that should be fixed before anything else.

### 1a. Destroy endpoint doesn't check account ownership

`src/mshkn/api/computers.py:262-270` — `destroy_computer` ignores the `account` parameter. Any valid API key can destroy any computer.

**Fix:** Check `computer.account_id == account.id` before destroying.

**Tests:** T8.2 (cross-tenant isolation)

### 1b. VMManager.initialize() doesn't recover system state

`src/mshkn/vm/manager.py:40-47` — Uses `list_computers_by_account(db, "%")` which doesn't match any real account. After restart, slot/volume counters reset, causing conflicts with existing tap devices and dm-thin volumes.

**Fix:** Query all running computers (add a `list_all_computers` DB function), or scan actual system state (tap devices, dmsetup ls).

**Tests:** T6.1 (crash recovery)

### 1c. No concurrent slot/volume allocation protection

`_allocate_slot()` and `_allocate_volume_id()` are plain increments with no locking. Concurrent creates could get the same slot.

**Fix:** Use asyncio.Lock or atomic DB counter.

**Tests:** T5.3 (concurrent creates/destroys)

## Priority 2: Caddy Dynamic Routing (days)

The proxy stub exists (`src/mshkn/proxy/caddy.py`) but isn't wired up. Caddy needs to be installed on the server and configured to route `{port}-{computer_id}.mshkn.dev` → VM internal IP.

**What's needed:**
- Wildcard DNS for `*.mshkn.dev` pointing to server
- Wildcard TLS cert (Caddy can do this with Cloudflare DNS challenge)
- CaddyClient wired into VMManager.create/destroy to add/remove routes
- Reverse proxy config: `{port}-{id}.mshkn.dev` → `http://{vm_ip}:{port}`

**Tests:** T4.1-T4.5 (auto-HTTPS, multiple ports, WebSocket, URL lifecycle, isolation)

## Priority 3: Nix Capability System (weeks)

The biggest chunk of unimplemented work. The stubs exist (`src/mshkn/capability/`) but don't do anything.

**What's needed:**
- Nix expressions that build capability layers (python-3.12 with packages, node-22, ffmpeg, etc.)
- `CapabilityResolver` that takes `uses: ["python-3.12(numpy)"]` and produces a Nix closure
- `CapabilityBuilder` that builds the closure and creates an overlayfs layer
- `CapabilityCache` that caches built layers on NVMe
- Host-side overlayfs composition: base image + capability layers → rootfs
- Package manager interception: wrapper scripts for pip/npm/apt that return structured errors with `suggested_action`
- Integration with VMManager.create to apply layers before boot

**Tests:** T3.1-T3.9 (all capability tests), T10.1-T10.2 (agent workflows needing capabilities)

**Dependency:** None. Can be done in parallel with merge.

---

## Priority 4: Merge End-to-End (weeks)

The 3-way merge algorithm exists (`src/mshkn/checkpoint/merge.py`) but isn't wired to the API. The merge endpoint returns a placeholder.

**What's needed:**
- Mount disk volumes for parent, fork_a, fork_b (dm-thin activate + mount)
- Run `three_way_merge` on mounted filesystems
- Create new checkpoint from merged result
- Handle conflicts: store conflict info, expose via API, allow resolution
- `checkpoint_resolve_conflicts` endpoint (doesn't exist yet)
- Upload merged checkpoint to R2

**Tests:** T2.2-T2.7, T2.9-T2.10 (all merge correctness tests), T1.5 (merge latency)

**Dependency:** None. Can be done in parallel with capabilities.
